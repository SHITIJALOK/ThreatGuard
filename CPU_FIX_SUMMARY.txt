TECHNICAL EXPLANATION:
======================

The "Not Responding" happened because:

1. Packet arrives (e.g., port scan)
2. Flow times out and gets classified
3. ML model.predict() called (fast, ~10ms)
4. If malicious → _ensure_firewall_block() called
5. subprocess.run() starts → WAITS FOR NETSH (slow!)
6. netsh.exe adds firewall rule (50-500ms on average)
7. Return from subprocess.run() after 50-500ms
8. Meanwhile: More packets queuing up
9. Scapy sniff() waiting for _process_packet callback
10. But main thread is BLOCKED on subprocess.run()
11. Queue backs up → 100+ packets waiting
12. Windows: "ThreatGuard (Not Responding)"

Now with the fix:

1-4. Same as above
5. _ensure_firewall_block() QUEUES rule in background thread
6. Returns IMMEDIATELY to caller
7. sniff() continues capturing packets
8. Background thread applies rule when ready (no impact)
9. Packet capture continues uninterrupted
10. UI stays responsive


Why heuristics are better:

  Before: ML model thinks about it (30-50ms)
  After: Check SYN ratio in 1ms
  
  Heuristic: SYN packets > 50% + no ACKs = Port Scan

DoS/Flood Detection:
  Before: ML model thinks about it (30-50ms)
  After: Check packets/sec in <1ms
  
  Heuristic: 100+ pps on single flow = DoS
  Also a SIGNATURE (guaranteed match)

Result: Detect obvious attacks instantly, use ML for edge cases
